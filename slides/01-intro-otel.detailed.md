## О воркшопе

OpenTelemetry становится стандартом для сбора телеметрии, но как применить его к PostgreSQL в production? В этом воркшопе мы разберём pgpro-otel-collector — готовый сборщик метрик и журналов от Postgres Professional.

На практике вы развернёте полноценный стек мониторинга: от настройки агента до дашбордов в Grafana. Узнаете не только «как сделать», но «на что смотреть» и «как интерпретировать данные».

**Для кого:** администраторы PostgreSQL, системные администраторы, DevOps — все, кто хочет перейти от теории Observability к работающей системе мониторинга.

**Чему вы научитесь:**

1. **Архитектура** — зачем нужен otel-collector и его преимущества
2. **Сбор данных** — метрики PostgreSQL (активность, ожидания, репликация) и структурированные логи
3. **Pipeline** — метрики в VictoriaMetrics, логи в VictoriaLogs
4. **Визуализация** — дашборды в Grafana для оперативного контроля и анализа проблем

**Формат:** интенсивный воркшоп (~4 часа). Шаг за шагом от нуля до работающей системы. К концу сессии — готовый конфигурационный набор для адаптации в своей инфраструктуре.

**Требования:** базовый опыт администрирования PostgreSQL, знакомство с Linux и Docker. Желательно понимание Prometheus и Grafana.

## Что такое Observability?

Observability (наблюдаемость) — это способность понять внутреннее состояние системы по её внешним сигналам. В отличие от классического мониторинга, где мы заранее решаем «что проверять», observability позволяет задавать произвольные вопросы о поведении системы и получать ответы без изменения кода или конфигурации.

Три столпа наблюдаемости:

- **Метрики** — числовые измерения, собираемые с определённым интервалом. Показывают текущее состояние и тренды: загрузку CPU, количество соединений, скорость транзакций, объём WAL. Компактны в хранении, идеальны для алертов и дашбордов.

- **Логи** — записи о дискретных событиях в системе. Содержат контекст: кто, когда, что произошло. В PostgreSQL это ошибки, checkpoints, медленные запросы, блокировки. Незаменимы при расследовании инцидентов.

- **Трейсы** — путь запроса через компоненты системы. Показывают, где именно запрос провёл время. Полезны в микросервисных архитектурах, где запрос проходит через несколько сервисов.

В этом воркшопе мы сосредоточимся на **метриках** и **логах** PostgreSQL — двух сигналах, которые дают максимум информации о здоровье и производительности СУБД.

## Зачем нужен OpenTelemetry?

**Проблема:** за десятилетия развития мониторинга индустрия накопила множество несовместимых инструментов. Каждый — со своим форматом данных, протоколом передачи и агентом, который нужно устанавливать и настраивать отдельно. При смене хранилища приходится менять агенты и переписывать интеграции.

- **Prometheus** — собственный формат экспозиции метрик (text exposition format), pull-модель сбора через HTTP-скрейпинг
- **Elasticsearch / ELK** — JSON-документы, собственный протокол приёма через Logstash или Beats, свои агенты (Filebeat, Metricbeat)
- **Jaeger, Zipkin** — разные форматы span-ов для трейсов, несовместимые SDK для инструментации приложений
- **Zabbix** — собственный бинарный протокол между агентом и сервером, своя система шаблонов и discovery
- **Datadog, New Relic** — проприетарные агенты и API, vendor lock-in при переходе на другое решение
- **Graphite, InfluxDB** — свои протоколы приёма метрик (plaintext, line protocol), отдельные коллекторы (collectd, Telegraf)

В итоге на одном сервере могут работать 3-4 разных агента, каждый со своим конфигом и потреблением ресурсов.

**Решение:** OpenTelemetry — единый открытый стандарт для сбора, обработки и передачи всех типов телеметрии. Один агент (Collector), один протокол (OTLP), одна конфигурация — а данные отправляются в любое хранилище.

---

## OpenTelemetry — ключевые концепции

- **Сигналы** — три типа телеметрии: метрики, логи и трейсы. OpenTelemetry определяет единую модель данных для каждого типа, что позволяет обрабатывать и передавать их одинаковым образом вне зависимости от источника.

- **OTLP** (OpenTelemetry Protocol) — единый протокол передачи всех сигналов. Работает поверх gRPC или HTTP, поддерживает бинарный (protobuf) и JSON-форматы. Принимается большинством современных хранилищ: VictoriaMetrics, Grafana Tempo, Datadog и другими.

- **SDK** — библиотеки для инструментации приложений на разных языках (Go, Java, Python, .NET и др.). Позволяют приложению генерировать метрики, логи и трейсы и отправлять их по OTLP. В нашем воркшопе SDK не используется — мы собираем данные извне.

- **Collector** — агент, который собирает, обрабатывает и экспортирует телеметрию. Работает как отдельный процесс, не требует изменений в приложениях. Конфигурируется через YAML и состоит из трёх типов компонентов: receivers, processors, exporters.

## OpenTelemetry Collector — архитектура

```
  ┌────────────────────────────────────────────────────────┐
  │               OpenTelemetry Collector                   │
  │                                                        │
  │  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐ │
  │  │  Receivers   │─▸│  Processors  │─▸│  Exporters   │───▸ Хранилища
  │  │              │  │              │  │              │ │
  │  │ postgrespro  │  │ batch        │  │ otlphttp     │ │
  │  │ hostmetrics  │  │ filter       │  │ prometheus   │ │
  │  │ filelog      │  │ transform    │  │ file         │ │
  │  └──────────────┘  └──────────────┘  └──────────────┘ │
  │                                                        │
  └────────────────────────────────────────────────────────┘
```

- **Receivers** — точки входа данных в коллектор. Receiver подключается к источнику (база данных, файл логов, HTTP-эндпоинт) и преобразует данные во внутренний формат OpenTelemetry. Каждый receiver работает независимо и может обслуживать свой тип сигнала.

- **Processors** — промежуточная обработка данных между приёмом и отправкой. Позволяют фильтровать ненужные метрики, добавлять атрибуты (например, имя сервиса), группировать данные в пакеты (batch), ограничивать потребление памяти. Процессоры выполняются последовательно в порядке, указанном в конфигурации.

- **Exporters** — точки выхода данных из коллектора. Отправляют обработанные данные во внешние системы хранения и визуализации. Один и тот же тип данных можно отправить в несколько хранилищ одновременно — например, метрики и в Prometheus, и в VictoriaMetrics.

---

## Receivers — источники данных

Receivers определяют, откуда коллектор получает данные. Каждый receiver реализует конкретный протокол или механизм сбора. Можно использовать несколько receivers одновременно — например, собирать метрики из PostgreSQL и ОС, а логи из файлов.

| Receiver | Что собирает |
|----------|-------------|
| **`postgrespro`** | Метрики PostgreSQL — активность, WAL, блокировки, кэш и др. |
| `hostmetrics` | Метрики ОС — CPU, память, диск, сеть, load average |
| `filelog` | Логи из файлов — поддержка JSON-парсинга, multiline, severity mapping |
| `otlp` | Приём данных по протоколу OTLP (gRPC/HTTP) от приложений с OpenTelemetry SDK |
| `prometheus` | Скрейпинг Prometheus-эндпоинтов — совместимость с существующими экспортерами |
| `journald` | Логи из systemd journal — альтернатива filelog для систем с systemd |
| `sqlquery` | Выполнение произвольных SQL-запросов для сбора метрик и логов из баз данных |

Полный каталог receivers: [opentelemetry-collector-contrib/receiver](https://github.com/open-telemetry/opentelemetry-collector-contrib/tree/main/receiver)

---

## Processors — обработка данных

Processors обрабатывают данные между приёмом и отправкой. Выполняются последовательно в порядке, указанном в конфигурации pipeline. Позволяют трансформировать, фильтровать и обогащать телеметрию, не меняя источник данных.

| Processor | Что делает |
|-----------|-----------|
| `batch` | Группирует данные в пакеты для эффективной отправки — снижает нагрузку на сеть |
| `memory_limiter` | Ограничивает потребление памяти коллектором — защита от OOM |
| `resource` | Добавляет или изменяет атрибуты ресурса (service.name, host.name и т.д.) |
| `attributes` | Преобразует атрибуты записей — переименование, удаление, конвертация типов |
| `filter` | Фильтрует данные по условиям — drop или include по имени метрики, severity лога |
| `transform` | Трансформация через OTTL-выражения — мощный язык для произвольных преобразований |
| `k8sattributes` | Обогащает данные атрибутами Kubernetes — pod, namespace, deployment |

Полный каталог processors: [opentelemetry-collector-contrib/processor](https://github.com/open-telemetry/opentelemetry-collector-contrib/tree/main/processor)

---

## Exporters — назначения

Exporters отправляют обработанные данные во внешние системы. Один pipeline может содержать несколько exporters — данные будут отправлены в каждый из них параллельно. Это позволяет одновременно писать, например, в долгосрочное хранилище и в систему алертинга.

| Exporter | Куда отправляет |
|----------|----------------|
| `otlphttp` | По протоколу OTLP через HTTP — универсальный exporter для любых OTLP-совместимых хранилищ |
| `otlp` | По протоколу OTLP через gRPC — более эффективен для больших объёмов данных |
| `prometheus` | Публикует метрики на HTTP-эндпоинте в формате Prometheus — для скрейпинга |
| `file` | Записывает данные в файл — полезно для отладки и аудита |
| `kafka` | Отправка в Apache Kafka — для построения сложных pipeline с потоковой обработкой |
| `debug` | Выводит данные в лог коллектора — для отладки конфигурации на этапе настройки |
| `loadbalancing` | Распределяет данные между несколькими бэкендами — горизонтальное масштабирование |

Полный каталог exporters: [opentelemetry-collector-contrib/exporter](https://github.com/open-telemetry/opentelemetry-collector-contrib/tree/main/exporter)

## Pipelines — связываем всё вместе

Pipeline (пайплайн) — это цепочка, которая соединяет receivers, processors и exporters в единый поток обработки данных. Каждый пайплайн обслуживает один тип сигнала: метрики, логи или трейсы. В одном коллекторе может работать несколько пайплайнов одновременно.

```yaml
service:
  pipelines:
    metrics:                      # пайплайн для метрик
      receivers:  [postgrespro, hostmetrics]
      processors: [memory_limiter]
      exporters:  [otlphttp, prometheus]

    logs:                         # пайплайн для логов
      receivers:  [filelog]
      processors: [resource, batch]
      exporters:  [otlphttp/victorialogs]
```

В нашем воркшопе два пайплайна: один для метрик PostgreSQL и ОС, второй для JSON-логов PostgreSQL. Метрики и логи собираются, обрабатываются и отправляются независимо друг от друга.

| PGPRO OTEL Collector  | 8888, 8889 | Центральный компонент: собирает метрики PostgreSQL и ОС через receivers, обрабатывает через processors и отправляет в хранилища через exporters |

## Итоги: введение в OpenTelemetry

- **OpenTelemetry** — открытый стандарт, объединяющий сбор метрик, логов и трейсов в единую модель данных и единый протокол (OTLP)
- **Collector** — универсальный агент, заменяющий множество разрозненных инструментов. Один процесс, одна конфигурация, любые источники и хранилища
- **Архитектура Receiver → Processor → Exporter** позволяет гибко комбинировать компоненты и строить pipeline любой сложности
- **PGPRO OTEL Collector** — специализированная сборка с нативным receiver для PostgreSQL, которую мы будем использовать на практике
- **Стек воркшопа**: PostgreSQL → PGPRO OTEL Collector → VictoriaMetrics + VictoriaLogs → Grafana — полный путь от сбора данных до визуализации
